    27вҶ’    27вҶ’## Overview
    28вҶ’    28вҶ’
    29вҶ’    29вҶ’Faya POS is the payment-taking app for the Faya ecosystem. It is **strictly a mobile application** вҖ” designed for phones and physical POS terminals. Merchants bind the app to their business account, admins approve devices, and staff use unique PIN codes to process payments.
    30вҶ’    30вҶ’
    31вҶ’    31вҶ’Key principles:
    32вҶ’    32вҶ’
    33вҶ’    33вҶ’- **No lock screen** вҖ” The POS is always visible. Staff enter their PIN only when they need to do something (collect payment, refund, switch staff).
    34вҶ’    34вҶ’- **Auto-detect payment method** вҖ” No manual method selection. The system detects NFC, chip, or swipe automatically based on device capabilities.
    35вҶ’    35вҶ’- **Every transaction is tied to staff** вҖ” All payments and refunds are logged with the staff member who processed them, and sent to the merchant.
    36вҶ’    36вҶ’- **Device binding requires admin approval** вҖ” A merchant can bind a device, but it only becomes active after an admin approves it from the Merchant App.
    37вҶ’    37вҶ’- **Device check is mandatory** вҖ” If the device fails capability checks (no NFC, no network), the merchant cannot log in at all.

   390вҶ’   390вҶ’## API Reference
   391вҶ’   391вҶ’
   392вҶ’   392вҶ’### Merchant Login
   393вҶ’   393вҶ’```
   394вҶ’   394вҶ’POST /api/pos/merchant/login
   395вҶ’   395вҶ’Body: { email, password, deviceCapabilities, systemInfo }
   396вҶ’   396вҶ’Response: { merchant, branch, device }
   397вҶ’   397вҶ’```
   398вҶ’   398вҶ’Authenticates the merchant, creates/updates the device record with detected capabilities, and returns the device's current approval status. Device starts as `pending` unless already approved.
   399вҶ’   399вҶ’
   400вҶ’   400вҶ’### Staff PIN Login
   401вҶ’   401вҶ’```
   402вҶ’   402вҶ’POST /api/pos/auth/login
   403вҶ’   403вҶ’Body: { pin }
   404вҶ’   404вҶ’Response: { staff, merchant, branch, device, session }
   405вҶ’   405вҶ’```
   406вҶ’   406вҶ’Verifies staff by their unique PIN. Creates a new `PosSession`, ends any previous active session on the device (allows seamless staff switching). Returns staff details with role permissions.
   407вҶ’   407вҶ’
   408вҶ’   408вҶ’### Device Binding
   409вҶ’   409вҶ’```
   410вҶ’   410вҶ’POST /api/pos/device/bind
   411вҶ’   411вҶ’Body: { deviceId, merchantId, branchId }
   412вҶ’   412вҶ’Response: { approvalStatus, message }
   413вҶ’   413вҶ’```
   414вҶ’   414вҶ’Confirms the device binding after merchant login. Records the binding event with all device capabilities. Device remains pending until admin approves.
   415вҶ’   415вҶ’
   416вҶ’   416вҶ’### Device Approval
   417вҶ’   417вҶ’```
   418вҶ’   418вҶ’POST /api/pos/device/approve
   419вҶ’   419вҶ’Body: { deviceId }
   420вҶ’   420вҶ’Response: { approvalStatus, status, message }
   421вҶ’   421вҶ’```
   422вҶ’   422вҶ’Approves a pending device. Updates `approvalStatus` to `approved` and `status` to `active`. Creates an audit event. In production, this requires admin authentication.
   423вҶ’   423вҶ’
   424вҶ’   424вҶ’### Check Approval Status
   425вҶ’   425вҶ’```
   426вҶ’   426вҶ’GET /api/pos/device/approval-status?deviceId=xxx
   427вҶ’   427вҶ’Response: { approvalStatus, status }
   428вҶ’   428вҶ’```
   429вҶ’   429вҶ’Polled by the Awaiting Approval screen every 10 seconds.
   430вҶ’   430вҶ’
   431вҶ’   431вҶ’### Create Transaction
   432вҶ’   432вҶ’```
   433вҶ’   433вҶ’POST /api/pos/transactions
   434вҶ’   434вҶ’Body: { merchantId, branchId, staffId, deviceId, amount, currency, description, paymentMethod }
   435вҶ’   435вҶ’Response: Transaction object with reference
   436вҶ’   436вҶ’```
   437вҶ’   437вҶ’Creates a new transaction with `created` status. Generates a unique reference (`FAYA-2026-XXXXXX`) and idempotency key. Validates staff and device are active.
   438вҶ’   438вҶ’
   439вҶ’   439вҶ’### Finalize Transaction
   440вҶ’   440вҶ’```
   441вҶ’   441вҶ’POST /api/pos/transactions/:id/finalize
   442вҶ’   442вҶ’Body: { status, authCode, responseCode, cardScheme, maskedPan, ... }
   443вҶ’   443вҶ’Response: Updated transaction
   444вҶ’   444вҶ’```
   445вҶ’   445вҶ’Updates the transaction with the final payment result (approved/declined/failed) and card details.
   446вҶ’   446вҶ’
   447вҶ’   447вҶ’### Cancel Transaction
   448вҶ’   448вҶ’```
   449вҶ’   449вҶ’POST /api/pos/transactions/:id/cancel
   450вҶ’   450вҶ’Body: { reason }
   451вҶ’   451вҶ’```
   452вҶ’   452вҶ’Cancels a pending transaction.
   453вҶ’   453вҶ’
   454вҶ’   454вҶ’### Request Refund
   455вҶ’   455вҶ’```

   470вҶ’   470вҶ’
   471вҶ’   471вҶ’The app uses a single Zustand store (`pos-store.ts`) that manages:
   472вҶ’   472вҶ’
   473вҶ’   473вҶ’| State | Type | Description |
   474вҶ’   474вҶ’|---|---|---|
   475вҶ’   475вҶ’| `currentScreen` | `Screen` (18 screen types) | Active screen for the router |
   476вҶ’   476вҶ’| `previousScreen` | `Screen \| null` | For back navigation |
   477вҶ’   477вҶ’| `merchant` | `MerchantInfo \| null` | Bound merchant account |
   478вҶ’   478вҶ’| `branch` | `BranchInfo \| null` | Merchant's branch |
   479вҶ’   479вҶ’| `device` | `DeviceInfo \| null` | This POS device with capabilities |
   480вҶ’   480вҶ’| `staff` | `StaffInfo \| null` | Currently authenticated staff (null = no staff) |
   481вҶ’   481вҶ’| `pinPurpose` | `'collect-payment' \| 'refund' \| 'override'` | Why the PIN is being entered |
   482вҶ’   482вҶ’| `paymentFlow` | `PaymentFlowState` | Amount, currency, method, current transaction |
   483вҶ’   483вҶ’| `deviceBound` | `boolean` | Whether merchant has bound this device |
   484вҶ’   484вҶ’| `deviceApproved` | `boolean` | Whether admin has approved this device |
   485вҶ’   485вҶ’| `dashboardStats` | `DashboardStats \| null` | Today's payment stats |
   486вҶ’   486вҶ’| `selectedTransaction` | `TransactionInfo \| null` | For detail/refund views |
   487вҶ’   487вҶ’
   488вҶ’   488вҶ’### Key Actions
   489вҶ’   489вҶ’
   490вҶ’   490вҶ’- **`navigate(screen)`** вҖ” Change the active screen with history
   491вҶ’   491вҶ’- **`goBack()`** вҖ” Navigate to previous screen
   492вҶ’   492вҶ’- **`setStaff(staff)`** вҖ” Set or clear the current staff member
   493вҶ’   493вҶ’- **`merchantLogout()`** вҖ” Full reset: unbinds device, clears all state, goes to merchant login
   494вҶ’   494вҶ’- **`resetPaymentFlow()`** вҖ” Clears amount, transaction, and payment state
   495вҶ’   495вҶ’
   496вҶ’   496вҶ’### Utility Functions
   497вҶ’   497вҶ’
   498вҶ’   498вҶ’- **`formatAmount(kobo, currency)`** вҖ” Converts kobo to formatted currency string (e.g., `вӮҰ1,500.00`)
   499вҶ’   499вҶ’- **`formatDate(iso)`** вҖ” Human-readable date/time

   668вҶ’   668вҶ’## Business Rules
   669вҶ’   669вҶ’
   670вҶ’   670вҶ’### Payment Rules
   671вҶ’   671вҶ’1. **All card payments require online authorization** вҖ” No offline mode
   672вҶ’   672вҶ’2. **No QR, bank transfer, or cash payments** вҖ” Card/NFC only
   673вҶ’   673вҶ’3. **No manual card entry** вҖ” Must be tapped, inserted, or swiped
   674вҶ’   674вҶ’4. **No screenshot proof** вҖ” All transactions verified by processor
   675вҶ’   675вҶ’5. **Full PAN, CVV, and PIN are never stored** вҖ” PCI-DSS compliance
   676вҶ’   676вҶ’
   677вҶ’   677вҶ’### Staff Permission Levels
   678вҶ’   678вҶ’| Action | Cashier | Manager | Admin |
   679вҶ’   679вҶ’|---|---|---|---|
   680вҶ’   680вҶ’| Process payments | вң… | вң… | вң… |
   681вҶ’   681вҶ’| View transaction history | вң… | вң… | вң… |
   682вҶ’   682вҶ’| Request refund | вқҢ | вң… | вң… |
   683вҶ’   683вҶ’| Override current staff | вқҢ | вң… | вң… |
   684вҶ’   684вҶ’| Device settings | вң… (view) | вң… | вң… |
   685вҶ’   685вҶ’| Unbind merchant | вқҢ | вқҢ | вң… |
   686вҶ’   686вҶ’
   687вҶ’   687вҶ’### Device Rules
   688вҶ’   688вҶ’1. A phone POS only has NFC вҖ” no chip or swipe capability
   689вҶ’   689вҶ’2. A physical terminal has at least 2 of (NFC, chip, swipe)
   690вҶ’   690вҶ’3. Device check must pass before merchant can log in
   691вҶ’   691вҶ’4. A device is only fully bound after admin approves it
   692вҶ’   692вҶ’5. When binding, all device information and capabilities are sent to the admin
   693вҶ’   693вҶ’6. A merchant can have multiple approved devices
   694вҶ’   694вҶ’
   695вҶ’   695вҶ’### Transaction Rules
   696вҶ’   696вҶ’1. Amounts are stored in **kobo** (smallest currency unit) вҖ” вӮҰ1 = 100 kobo
   697вҶ’   697вҶ’2. Every transaction is tied to a specific staff member
   698вҶ’   698вҶ’3. All transactions and refunds are reported to the merchant
   699вҶ’   699вҶ’4. Transaction references follow format: `FAYA-2026-XXXXXX`
   700вҶ’   700вҶ’5. Refunds can be full or partial, but always require a reason
   701вҶ’   701вҶ’

   704вҶ’   704вҶ’## Security Model
   705вҶ’   705вҶ’
   706вҶ’   706вҶ’### Authentication Layers
   707вҶ’   707вҶ’```
   708вҶ’   708вҶ’Layer 1: Device Check
   709вҶ’   709вҶ’  в””в”Җв–ә Hardware capabilities verified (NFC, network, integrity)
   710вҶ’   710вҶ’
   711вҶ’   711вҶ’Layer 2: Merchant Login
   712вҶ’   712вҶ’  в””в”Җв–ә Email/phone + password authentication
   713вҶ’   713вҶ’  в””в”Җв–ә Device bound to merchant account
   714вҶ’   714вҶ’  в””в”Җв–ә Admin must approve the device
   715вҶ’   715вҶ’
   716вҶ’   716вҶ’Layer 3: Staff PIN
   717вҶ’   717вҶ’  в””в”Җв–ә Unique PIN per staff member
   718вҶ’   718вҶ’  в””в”Җв–ә PIN only required for actions (payment, refund, switch)
   719вҶ’   719вҶ’  в””в”Җв–ә POS always visible вҖ” no lock screen
   720вҶ’   720вҶ’  в””в”Җв–ә Staff sessions tracked in PosSession table
   721вҶ’   721вҶ’```
   722вҶ’   722вҶ’
   723вҶ’   723вҶ’### Audit Trail
   724вҶ’   724вҶ’Every significant action creates a `PosDeviceEvent`:
   725вҶ’   725вҶ’- Device activated, approved, blocked
   726вҶ’   726вҶ’- Staff login, logout
   727вҶ’   727вҶ’- Transaction created, approved, declined, failed
   728вҶ’   728вҶ’- Receipt printed, shared
   729вҶ’   729вҶ’- Refund requested
   730вҶ’   730вҶ’- Device integrity failed
   731вҶ’   731вҶ’- NFC disabled
   732вҶ’   732вҶ’- Config sync failed
   733вҶ’   733вҶ’- App version blocked
   734вҶ’   734вҶ’
   735вҶ’   735вҶ’### Data Protection
   736вҶ’   736вҶ’- Card numbers are stored as masked PAN only (`**** **** **** 1234`)
   737вҶ’   737вҶ’- CVV is never stored
   738вҶ’   738вҶ’- PIN is stored in plain text (demo only вҖ” production must hash)
   739вҶ’   739вҶ’- Staff sessions use UUID tokens
   740вҶ’   740вҶ’- Idempotency keys prevent duplicate transactions
