    27→## Overview
    28→
    29→Faya POS is the payment-taking app for the Faya ecosystem. It is **strictly a mobile application** — designed for phones and physical POS terminals. Merchants bind the app to their business account, admins approve devices, and staff use unique PIN codes to process payments.
    30→
    31→Key principles:
    32→
    33→- **No lock screen** — The POS is always visible. Staff enter their PIN only when they need to do something (collect payment, refund, switch staff).
    34→- **Auto-detect payment method** — No manual method selection. The system detects NFC, chip, or swipe automatically based on device capabilities.
    35→- **Every transaction is tied to staff** — All payments and refunds are logged with the staff member who processed them, and sent to the merchant.
    36→- **Device binding requires admin approval** — A merchant can bind a device, but it only becomes active after an admin approves it from the Merchant App.
    37→- **Device check is mandatory** — If the device fails capability checks (no NFC, no network), the merchant cannot log in at all.

    81→## Device Types & Capabilities
    82→
    83→The app performs **real system checks** to determine what the device can do:
    84→
    85→| Device Type | NFC | Chip Reader | Swipe Reader | How Determined |
    86→|---|---|---|---|---|
    87→| **Phone POS** | ✅ | ❌ | ❌ | Mobile device with NFC (phone's built-in NFC) |
    88→| **Physical Terminal** | ✅ | ✅ | ✅ | Has 2+ of (NFC, chip, swipe) — typically via UA detection or capability count |
    89→| **Faya NFC** | ✅ | ❌ | ❌ | Dedicated Faya NFC device |
    90→
    91→### Detection Logic
    92→
    93→1. Check for physical terminal User-Agent patterns (Ingenico, Verifone, PAX, etc.)
    94→2. Check Web NFC API (`NDEFReader` in window)
    95→3. Count available payment methods:
    96→   - Only NFC + mobile → **Phone POS**
    97→   - 2+ methods → **Physical Terminal**
    98→   - Desktop browser demo → Simulates **Phone POS** with NFC
    99→4. If no payment method available → **Device check fails** → Login blocked

   390→## API Reference
   391→
   392→### Merchant Login
   393→```
   394→POST /api/pos/merchant/login
   395→Body: { email, password, deviceCapabilities, systemInfo }
   396→Response: { merchant, branch, device }
   397→```
   398→Authenticates the merchant, creates/updates the device record with detected capabilities, and returns the device's current approval status. Device starts as `pending` unless already approved.
   399→
   400→### Staff PIN Login
   401→```
   402→POST /api/pos/auth/login
   403→Body: { pin }
   404→Response: { staff, merchant, branch, device, session }
   405→```
   406→Verifies staff by their unique PIN. Creates a new `PosSession`, ends any previous active session on the device (allows seamless staff switching). Returns staff details with role permissions.
   407→
   408→### Device Binding
   409→```
   410→POST /api/pos/device/bind
   411→Body: { deviceId, merchantId, branchId }
   412→Response: { approvalStatus, message }
   413→```
   414→Confirms the device binding after merchant login. Records the binding event with all device capabilities. Device remains pending until admin approves.
   415→
   416→### Device Approval
   417→```
   418→POST /api/pos/device/approve
   419→Body: { deviceId }
   420→Response: { approvalStatus, status, message }
   421→```
   422→Approves a pending device. Updates `approvalStatus` to `approved` and `status` to `active`. Creates an audit event. In production, this requires admin authentication.
   423→
   424→### Check Approval Status
   425→```
   426→GET /api/pos/device/approval-status?deviceId=xxx
   427→Response: { approvalStatus, status }
   428→```
   429→Polled by the Awaiting Approval screen every 10 seconds.
   430→
   431→### Create Transaction
   432→```
   433→POST /api/pos/transactions
   434→Body: { merchantId, branchId, staffId, deviceId, amount, currency, description, paymentMethod }
   435→Response: Transaction object with reference
   436→```
   437→Creates a new transaction with `created` status. Generates a unique reference (`FAYA-2026-XXXXXX`) and idempotency key. Validates staff and device are active.
   438→
   439→### Finalize Transaction
   440→```
   441→POST /api/pos/transactions/:id/finalize
   442→Body: { status, authCode, responseCode, cardScheme, maskedPan, ... }
   443→Response: Updated transaction
   444→```
   445→Updates the transaction with the final payment result (approved/declined/failed) and card details.
   446→
   447→### Cancel Transaction
   448→```
   449→POST /api/pos/transactions/:id/cancel
   450→Body: { reason }
   451→```
   452→Cancels a pending transaction.
   453→
   454→### Request Refund
   455→```
   456→POST /api/pos/transactions/:id/refund-request
   457→Body: { amount?, reason, staffId }
   458→```
   459→Submits a refund request. Can be full or partial. Requires a reason. Sent to the merchant for review.
   460→
   461→### Dashboard Stats
   462→```
   463→GET /api/pos/dashboard?staffId=xxx&branchId=xxx
   464→Response: { totalApprovedAmount, approvedCount, declinedCount, paymentMethodBreakdown }
   465→```

   469→## State Management
   470→
   471→The app uses a single Zustand store (`pos-store.ts`) that manages:
   472→
   473→| State | Type | Description |
   474→|---|---|---|
   475→| `currentScreen` | `Screen` (18 screen types) | Active screen for the router |
   476→| `previousScreen` | `Screen \| null` | For back navigation |
   477→| `merchant` | `MerchantInfo \| null` | Bound merchant account |
   478→| `branch` | `BranchInfo \| null` | Merchant's branch |
   479→| `device` | `DeviceInfo \| null` | This POS device with capabilities |
   480→| `staff` | `StaffInfo \| null` | Currently authenticated staff (null = no staff) |
   481→| `pinPurpose` | `'collect-payment' \| 'refund' \| 'override'` | Why the PIN is being entered |
   482→| `paymentFlow` | `PaymentFlowState` | Amount, currency, method, current transaction |
   483→| `deviceBound` | `boolean` | Whether merchant has bound this device |
   484→| `deviceApproved` | `boolean` | Whether admin has approved this device |
   485→| `dashboardStats` | `DashboardStats \| null` | Today's payment stats |
   486→| `selectedTransaction` | `TransactionInfo \| null` | For detail/refund views |
   487→
   488→### Key Actions
   489→
   490→- **`navigate(screen)`** — Change the active screen with history
   491→- **`goBack()`** — Navigate to previous screen
   492→- **`setStaff(staff)`** — Set or clear the current staff member
   493→- **`merchantLogout()`** — Full reset: unbinds device, clears all state, goes to merchant login
   494→- **`resetPaymentFlow()`** — Clears amount, transaction, and payment state
   495→
   496→### Utility Functions
   497→
   498→- **`formatAmount(kobo, currency)`** — Converts kobo to formatted currency string (e.g., `₦1,500.00`)
   499→- **`formatDate(iso)`** — Human-readable date/time
   500→- **`getPaymentMethodLabel(code)`** — Maps method codes to readable labels
   501→- **`getStatusColor(status)`** — Returns Tailwind classes for transaction status badges
   502→- **`getRoleLabel(role)`** / **`getRoleColor(role)`** — Staff role display helpers

   506→## Screens
   507→
   508→### 1. Splash Screen
   509→Branded loading screen with Faya POS logo. Auto-navigates to Device Check after 2.5 seconds.
   510→
   511→### 2. Device Check Screen
   512→Runs real system checks in sequence:
   513→- App version compatibility
   514→- Network connectivity (online/offline)
   515→- NFC hardware availability
   516→- NFC enabled state
   517→- Chip reader (physical terminals only)
   518→- Swipe reader (physical terminals only)
   519→- Device integrity
   520→- Payment capability summary
   521→
   522→**On failure**: Shows a blocked state with "Retry Device Check" button. Merchant **cannot** login.
   523→
   524→### 3. Merchant Login Screen
   525→Email/phone + password form. Detects device capabilities and sends them with the login request. Displays the detected device type and supported payment methods.
   526→
   527→### 4. Device Binding Screen
   528→Shows a summary of:

   668→## Business Rules
   669→
   670→### Payment Rules
   671→1. **All card payments require online authorization** — No offline mode
   672→2. **No QR, bank transfer, or cash payments** — Card/NFC only
   673→3. **No manual card entry** — Must be tapped, inserted, or swiped
   674→4. **No screenshot proof** — All transactions verified by processor
   675→5. **Full PAN, CVV, and PIN are never stored** — PCI-DSS compliance
   676→
   677→### Staff Permission Levels
   678→| Action | Cashier | Manager | Admin |
   679→|---|---|---|---|
   680→| Process payments | ✅ | ✅ | ✅ |
   681→| View transaction history | ✅ | ✅ | ✅ |
   682→| Request refund | ❌ | ✅ | ✅ |
   683→| Override current staff | ❌ | ✅ | ✅ |
   684→| Device settings | ✅ (view) | ✅ | ✅ |
   685→| Unbind merchant | ❌ | ❌ | ✅ |
   686→
   687→### Device Rules
   688→1. A phone POS only has NFC — no chip or swipe capability
   689→2. A physical terminal has at least 2 of (NFC, chip, swipe)
   690→3. Device check must pass before merchant can log in
   691→4. A device is only fully bound after admin approves it
   692→5. When binding, all device information and capabilities are sent to the admin
   693→6. A merchant can have multiple approved devices
   694→
   695→### Transaction Rules
   696→1. Amounts are stored in **kobo** (smallest currency unit) — ₦1 = 100 kobo
   697→2. Every transaction is tied to a specific staff member
   698→3. All transactions and refunds are reported to the merchant
   699→4. Transaction references follow format: `FAYA-2026-XXXXXX`
   700→5. Refunds can be full or partial, but always require a reason

   704→## Security Model
   705→
   706→### Authentication Layers
   707→```
   708→Layer 1: Device Check
   709→  └─► Hardware capabilities verified (NFC, network, integrity)
   710→
   711→Layer 2: Merchant Login
   712→  └─► Email/phone + password authentication
   713→  └─► Device bound to merchant account
   714→  └─► Admin must approve the device
   715→
   716→Layer 3: Staff PIN
   717→  └─► Unique PIN per staff member
   718→  └─► PIN only required for actions (payment, refund, switch)
   719→  └─► POS always visible — no lock screen
   720→  └─► Staff sessions tracked in PosSession table
   721→```
   722→
   723→### Audit Trail
   724→Every significant action creates a `PosDeviceEvent`:
   725→- Device activated, approved, blocked
   726→- Staff login, logout
   727→- Transaction created, approved, declined, failed
   728→- Receipt printed, shared
   729→- Refund requested
   730→- Device integrity failed
   731→- NFC disabled
   732→- Config sync failed
   733→- App version blocked
   734→
   735→### Data Protection
   736→- Card numbers are stored as masked PAN only (`**** **** **** 1234`)
   737→- CVV is never stored
   738→- PIN is stored in plain text (demo only — production must hash)
   739→- Staff sessions use UUID tokens
   740→- Idempotency keys prevent duplicate transactions
